# To-Do

Список дел на Next.js

## Запуск
```bash
pnpm i
pnpm dev
```

http://localhost:3000

## Стек

- **Next.js 16** (App Router) + **React 19**
- **TypeScript**
- **Prisma 7** + **SQLite** (`@prisma/adapter-better-sqlite3`)
- **zod** — валидация и единый источник схем данных
- **@asteasolutions/zod-to-openapi** — генерация `openapi.json` из zod-схем
- **openapi-typescript** — генерация типов клиента из `openapi.json`
- **openapi-fetch** + **openapi-react-query** — типизированный API-клиент и хуки
- **TanStack React Query** — кэш и синхронизация состояния
- **Tailwind CSS 4**

## Об использовании ИИ
Сделал работу на совесть. 

Архитектуру и backend, и frontend проектировал сам; frontend собирал с помощью нейросети, чтобы ускорить разработку UI и больше времени уделить backend и архитектуре.

Основной упор был на backend и его работу с frontend: сквозная типизация, валидация данных и работа с бд.

## Что реализовано

- **REST API.** Два endpoint — 
`/api/todos` (`GET`, `POST`, `PATCH /{id}`, `DELETE /{id}`)
`/api/users` (`GET`, `POST`, `DELETE /{id}`), каждый маршрут описан отдельной zod-схемой и валидируется на входе.

- **Валидация и типизация** Один и тот же zod-контракт описывает вход/выход API, генерирует OpenAPI-схему и типы клиента — фронтенд не может обратиться к несуществующему полю или отправить невалидные данные незаметно для TypeScript.

- **Автосинхронизация состояния.** После любой мутации (create/update/delete) React Query сам инвалидирует и перезапрашивает список — руками состояние на клиенте нигде не дергается.

## UI
- **Пользователи.** Создание и удаление; при удалении пользователя удаляются все его задачи.
- **Задачи (CRUD).** Создание, редактирование, удаление; поля — название, описание, приоритет (`LOW`/`MEDIUM`/`HIGH`), срок выполнения, исполнитель, автор.
- **Статусы задачи.** Отметка «выполнено» и закрепление («pinned») переключаются отдельными действиями, без открытия формы редактирования.
- **Доска задач.** Задачи сгруппированы по колонкам-исполнителям + отдельная колонка «Не назначено»; закреплённые задачи всегда идут первыми в колонке.


## Структура

```
prisma/                          схема БД, миграции, seed
src/app/                         роуты Next.js (страницы + API)
src/app/(main)/components/       UI-композиция, специфичная для главной страницы (todo-board)
src/server/                      серверный слой: zod-схемы, сервисы (доступ к БД), OpenAPI-регистрация
src/entities/                    клиентские хуки, типы и UI сущностей todo/user
src/features/                    формы и действия с бизнес-логикой (create-user, todo-form)
src/widgets/                     переиспользуемые composition-блоки общих страниц (header, footer)
src/shared/                      переиспользуемые UI-примитивы и типизированный API-клиент
```

## Почему так, а не проще

- **Prisma + SQLite (better-sqlite3 adapter).** SQLite — бд, Prisma даёт миграции и типобезопасный доступ к БД.
- **Разделение front и Back** `server/` Папка server под FSD архитектуру для удобства и раздеения логики.
- **React Query.** Инвалидация кэша и синхронизация состояния после create/update/delete без ручного управления списком на клиенте.
